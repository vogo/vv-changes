# Code Review: Orchestrator Agent Implementation

## Summary

The implementation closely follows the design document and is generally well-structured. Five issues were identified and four were fixed directly. All unit and integration tests pass after the changes.

## Issues Found and Fixed

### 1. FIXED: Duplicate utility function `convertToAIModelMessages`

**File:** `vv/agents/orchestrator.go` (lines 179-186, original)

**Severity:** Low (code quality)

The `convertToAIModelMessages` function duplicates `schema.ToAIModelMessages` from the vage framework, which has the exact same logic. This violates the DRY principle and means if the framework function is updated, the local copy would diverge.

**Fix applied:** Removed `convertToAIModelMessages` and replaced its usage in `classifyTask` with `schema.ToAIModelMessages`. Updated the corresponding test to use the framework function.

### 2. FIXED: `RunStream` plan mode re-classifies the request (wasted LLM call)

**File:** `vv/agents/orchestrator.go` (line 130, original)

**Severity:** High (correctness + cost)

The original code:
```go
case "plan":
    return agent.RunToStream(ctx, o, req), nil
```

This calls `agent.RunToStream(ctx, o, req)` which invokes `o.Run(ctx, req)`, which calls `o.classifyTask()` again -- making a second LLM classification call for the same request. This wastes tokens/money and could produce a different classification result the second time (e.g., direct instead of plan), causing inconsistent behavior.

**Fix applied:** The plan branch now wraps `o.runPlan` with the already-parsed plan and classification usage via `agent.NewCustomAgent`, avoiding re-classification.

### 3. FIXED: `fallbackRunStream` calls back into orchestrator when `fallbackAgent` is nil

**File:** `vv/agents/orchestrator.go` (line 277, original)

**Severity:** Medium (robustness)

When `fallbackAgent == nil`, the original code did `agent.RunToStream(ctx, o, req)` which calls `o.Run()`, which would classify, fail, and hit `fallbackRun` which correctly returns an error for nil fallback. While not an infinite loop, it wastes an LLM call on classification before ultimately failing. Returning an error immediately is more correct and efficient.

**Fix applied:** Changed to `return nil, fmt.Errorf("orchestrator: no fallback agent available")` to fail fast.

### 4. FIXED: `extractJSON` brace-counting ignores string contents

**File:** `vv/agents/orchestrator.go` (lines 338-353, original)

**Severity:** Medium (correctness)

The brace-counting fallback in `extractJSON` did not account for `{` and `}` characters inside JSON string values. For example, if the LLM returned `{"description": "write func() { return }"}`, the function would stop at the first `}` inside the string, returning truncated JSON.

**Fix applied:** Added string-awareness to the brace counter: tracks whether we're inside a double-quoted string and handles backslash escapes. Added a test case for this scenario.

### 5. FIXED: Missing compile-time interface check for `PlanAggregator`

**File:** `vv/agents/orchestrator.go`

**Severity:** Low (safety)

The `PlanAggregator` implements `orchestrate.Aggregator` but had no compile-time check. If the `Aggregator` interface changes, this would only fail at runtime.

**Fix applied:** Added `_ orchestrate.Aggregator = (*PlanAggregator)(nil)` to the compile-time check block.

## Observations (no change required)

### A. `extractJSON` code fence matching does not handle all edge cases

The code fence extraction (lines 320-335) handles ` ```json ` and bare ` ``` ` fences, but not other language hints like ` ```JSON ` (uppercase) or ` ``` json ` (space). Given that the orchestrator prompt instructs the LLM to respond with "ONLY a JSON object" and the fallback brace-counting is robust, this is acceptable. The LLM rarely uses non-standard fencing.

### B. Summary node iteration order in `buildNodes`

In the summary node's `InputMapper` (line 413), the code iterates over `upstream map[string]*schema.RunResponse` with `range`, which has non-deterministic ordering in Go. The `PlanAggregator.Aggregate` method sorts keys, so the final summary is deterministic, but the summary node's input prompt will vary between runs. This is acceptable since the summary is generated by an LLM, but worth noting.

### C. Test coverage

The test suite covers the major paths well: direct dispatch, plan mode, fallback on LLM error, fallback on invalid JSON, enrichRequest, extractJSON, findTerminalNodes, aggregateUsage, and PlanAggregator. One area that could benefit from additional testing is error handling within `buildNodes` when a plan references a valid agent that then fails during DAG execution -- though the existing `TestOrchestratorAgent_Run_PlanMode` exercises the happy path adequately.
